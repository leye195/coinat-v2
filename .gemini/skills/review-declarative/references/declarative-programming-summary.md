## 한줄 요약

**선언적 프로그래밍은 도구나 문법의 문제가 아니라, "어떻게(How)"에서 "무엇(What)"으로, 절차에서 관계로 전환하는 사고방식의 문제다.**

## 핵심 내용 상세

### 1. 가장 흔한 착각: "함수로 감싸면 선언적"

- 절차적 동작을 함수로 추상화한다고 해서 선언적이 되는 것이 **아니다**.
- `map`, `filter`, `reduce` 같은 배열 메서드를 사용해도 내부가 절차적 사고라면 여전히 절차적이다.

**절차적 사고 (배열 메서드를 사용했지만):**

```typescript
// map 안에서 단계별로 가격을 계산 → 여전히 "어떻게"에 집중
items.map((item) => {
  let price = item.basePrice;
  if (item.discount) {
    price = price * (1 - item.discount);
  }
  price = price * 1.1;
  return { ...item, finalPrice: Math.round(price * 100) / 100 };
});
```

**선언적 사고 (비즈니스 관계에 집중):**

```typescript
// 각 변환이 나타내는 관계를 명확히 선언
items.map(applyDiscount).map(addTax).map(formatPrice).filter(hasValidPrice);
```

> 핵심 구분: 절차적 코드는 **"어떻게(How) 단계별로 실행할 것인가"**, 선언적 코드는 **"무엇을(What) 원하는 관계인가"**에 집중한다.

---

### 2. 수학적 관점: 선언 = 불변적 관계

- `f(x) = 2x + 1`은 "x에 2를 곱하고 1을 더하라"는 **명령이 아니라**, x와 f(x) 사이의 **관계 선언**이다.
- 이 관계는 시간과 무관한 **영원한 진리**: x가 3이면 f(x)는 항상 7이다.
- `sin(x)` 역시 단위원에서 각도와 y좌표 사이의 관계를 선언한 것이지, 계산 방법을 지시한 것이 아니다.

> 프로그래밍에서 선언적 코드란 "이 문제의 본질은 이런 관계에 있다"는 것을 표현하는 것이다.

---

### 3. 절차적 vs 선언적 사고방식

**절차적 — 시간의 흐름 속 변화에 집중:**

```javascript
function calculateTotalPrice(items) {
  let total = 0;
  for (let i = 0; i < items.length; i++) {
    if (item.quantity > 0) {
      let itemPrice = item.price * item.quantity;
      if (item.discount) {
        itemPrice -= itemPrice * item.discount;
      }
      total += itemPrice;
    }
  }
  return total;
}
```

**선언적 — 논리적 관계에 집중:**

```javascript
const calculateTotalPrice = (items) =>
  items.filter(hasValidQuantity).map(calculateItemPrice).reduce(sum, 0);
```

**왜 관계적 사고가 중요한가?**

인간의 단기 기억은 동시에 **7±2개의 정보 단위**만 처리 가능하다(조지 밀러). 절차적 사고에서는 변수 10개면 2^10 = 1,024가지 상태 조합을 추적해야 하지만, 관계적 사고에서는 `h(x) = f(g(x))`처럼 각 함수의 입출력 관계만 이해하면 된다.

---

### 4. JSX가 선언적인 진짜 이유

"JSX를 쓰면 선언적" → "왜 선언적인가?" → "JSX를 쓰면 선언적이니까" 라는 **순환논법**에 빠지는 경우가 많다.

**JSX가 선언적인 진짜 이유:**

1. **구조적 관계 표현**: "UserCard는 Avatar와 UserInfo로 구성된다"는 관계를 선언. 수학에서 `A = {a, b, c}`라는 집합 관계를 선언하는 것과 동일.
2. **데이터-UI 대응 관계**: `todos.map(todo => <TodoItem ... />)`에서 데이터의 구조가 곧 UI의 구조가 된다.
3. **시간적 순서 무관**: 명령형에서는 `appendChild` 호출 순서가 결과를 바꾸지만, JSX에서는 포함 관계와 계층 구조만 중요.

---

### 5. 선언적과 절차적은 상대적이다 (핵심 통찰)

**같은 코드도 관찰하는 추상화 레벨에 따라 선언적/절차적이 달라진다.**

- **앱 개발자 관점**: `useQuery(['user', userId], () => fetchUser(userId))` → 완전히 선언적
- **라이브러리 내부**: QueryClient의 캐시 확인, 네트워크 요청, 재시도 로직 → 절차적
- **비즈니스 로직**: `processOrder` → 선언적 ("검증 → 결제 → 배송"이라는 관계)
- **인프라 레벨**: `processPayment` 내부 → 절차적 (API 호출, DB 저장, 에러 처리)

> "절차적 코드는 나쁘고 무조건 피해야 할 것"이라는 편견도 잘못된 생각이다. 가장 우아한 함수형 코드도 결국 CPU의 절차적 명령어로 실행된다.

**적절한 레벨 분리가 핵심:**

- 비즈니스 레벨 → 선언적 (도메인의 본질적 관계 표현)
- 인프라 레벨 → 절차적이어도 OK (효율성과 안전한 구현이 우선)

---

### 6. 착각하는 코드 vs 진짜 선언적인 코드

비동기 데이터 상태 관리를 예로 든 핵심 비교:

**착각 — 독립적인 boolean 상태들:**

```typescript
const [data, setData] = useState<T | null>(null);
const [loading, setLoading] = useState(false);
const [error, setError] = useState<string | null>(null);
// 문제: { data: someData, loading: true, error: "Network Error" } 같은
// 불가능한 상태가 타입 수준에서 허용됨
```

**진짜 선언적 — Union Type으로 상태 관계 선언:**

```typescript
type AsyncDataState<T> =
  | { status: "idle" }
  | { status: "loading" }
  | { status: "success"; data: T }
  | { status: "error"; error: string };
// "이 네 가지 상태 중 정확히 하나만 존재할 수 있다"는 불변적 관계를 선언
```

**선언적 접근의 이점:**

- 불가능한 상태를 **컴파일 타임에 원천 차단**
- TypeScript의 exhaustive check로 **모든 케이스 처리 강제**
- 새로운 상태 추가 시 **컴파일 에러로 누락 방지**
- 확장 시 타입 정의만 추가하면 됨 (boolean 조합의 기하급수적 증가 없음)

---

### 7. 실무 적용 가이드라인

| 상황                      | 접근법        | 이유                            |
| ------------------------- | ------------- | ------------------------------- |
| 비즈니스 로직 / 상태 관리 | **선언적**    | 불가능한 상태 차단, 의도 명확화 |
| UI 구조 표현              | **선언적**    | 데이터-UI 대응 관계 직관적 표현 |
| 데이터 변환 파이프라인    | **선언적**    | 각 단계 독립적, 테스트 용이     |
| 인프라 / 성능 최적화      | **절차적 OK** | 효율성과 성능이 우선            |
| 복잡한 에러 처리 / 재시도 | **절차적 OK** | 세밀한 제어가 필요              |
| 라이브러리 내부 구현      | **절차적 OK** | 캐시, 상태 업데이트 등 최적화   |

## 분석 및 시사점

### 글의 강점

1. **순환논법 문제 지적**: "선언적 도구를 쓰면 선언적"이라는 흔한 오류를 정확히 짚음
2. **수학적 근거**: 함수를 "계산 알고리즘"이 아닌 "구조 간의 대응 관계"로 보는 관점 제시
3. **상대성 개념**: 선언적/절차적이 절대적 구분이 아니라 추상화 레벨에 따른 상대적 개념임을 강조
4. **실용적 균형**: "모든 것을 선언적으로 만들 필요 없다"는 현실적 가이드라인 제시
5. **TypeScript 활용**: Union Type + exhaustive check를 통한 컴파일 타임 안전성 시연

### 실무 적용 포인트

- **상태 모델링**: boolean 플래그 여러 개 대신 discriminated union 사용
- **함수 설계**: 내부 구현이 아닌 입출력 관계가 명확한 이름과 시그니처 부여
- **추상화 레벨 인식**: 현재 작업하는 레벨이 비즈니스인지 인프라인지 구분하고 적절한 접근법 선택
- **컴포넌트 설계**: 데이터 구조와 UI 구조의 대응 관계가 직관적으로 읽히도록 설계

### 한계 / 보완할 점

- 함수형 프로그래밍과 선언적 프로그래밍의 관계에 대한 더 깊은 논의가 있으면 좋겠음
- side effect 관리 (IO 모나드 등)에 대한 언급이 부족
- 실무에서 "적절한 추상화 레벨"을 판단하는 구체적 기준이 다소 모호

---

## 핵심 인용

> "선언적 프로그래밍의 핵심은 함수 사용 여부가 아니라, 그 함수가 어떤 추상화를 제공하느냐에 따라 달라진다."

> "선언적 프로그래밍은 도구가 아니라 사고방식이며, 문법이 아니라 철학이다."

> "코드는 컴퓨터를 위한 것이 아니라 사람을 위한 것이다."
