# 선언적 vs 절차적 코드 분석

에반문의 "선언적 프로그래밍에 대한 착각과 오해" 분석 프레임워크를 기반으로 코드를 평가하고 개선 의견을 제시한다.

핵심 전제: **선언적 프로그래밍은 도구나 문법이 아니라 사고방식의 문제다.** `map`/`filter`를 쓴다고 선언적이 되는 것이 아니며, 절차적 코드가 적절한 레벨도 있다.

## Step 1: 대상 파일 결정

`$ARGUMENTS`를 파싱하여 분석 대상과 옵션을 결정한다.

### 분석 대상

| 인자 형태                               | 동작                                                                         |
| --------------------------------------- | ---------------------------------------------------------------------------- |
| 파일 경로 (`src/pages/Foo.tsx`)         | 해당 파일을 읽어 분석                                                        |
| glob 패턴 (`src/pages/Market/**/*.tsx`) | 매칭되는 파일들을 읽어 분석                                                  |
| `--diff`                                | `git diff --name-only HEAD`로 변경된 `.ts`/`.tsx` 파일을 대상으로 분석       |
| `--diff --staged`                       | `git diff --cached --name-only`로 staged된 `.ts`/`.tsx` 파일을 대상으로 분석 |
| 비어있음                                | staged + modified 파일 중 `.ts`/`.tsx` 파일을 대상으로 분석                  |

`--diff` 모드에서는 변경된 파일 전체를 읽되, diff 부분에 가중치를 두어 분석한다.

### 출력 옵션

| 옵션                 | 동작                                               |
| -------------------- | -------------------------------------------------- |
| (없음)               | 터미널에 분석 결과 출력                            |
| `--save <path>`      | 지정 경로에 `.md` 파일로 저장하고, 터미널에도 출력 |
| `--save` (경로 없음) | `./declarative-review-{YYYY-MM-DD}.md`로 저장      |

## Step 2: 코드 읽기

대상 파일들을 Read 도구로 읽는다. 파일이 많은 경우 주요 로직 파일(페이지, 훅, 유틸)을 우선한다.

## Step 3: 5가지 기준으로 분석

### 기준 1: 관계 vs 절차 (핵심)

코드가 "어떻게(How) 단계별로 실행하는가"에 집중하는지, "무엇을(What) 원하는 관계인가"에 집중하는지 판단한다.

**절차적 신호:**

- 시간적 순서에 의존하는 로직 ("먼저 A를 하고, 그 다음 B를 하고...")
- `let` 변수를 선언하고 단계적으로 변이(mutation)하는 패턴
- `for`/`while` 루프 안에서 상태를 누적하는 패턴
- 함수 내부에서 여러 단계의 조건 분기와 상태 변경이 혼재

**선언적 신호:**

- 데이터 변환의 관계를 체이닝이나 합성으로 표현
- 각 변환 단계가 독립적인 함수로 분리되어 비즈니스 의도가 이름에 드러남
- 입력과 출력 사이의 매핑 관계가 명확

**판단 체크리스트:**

- [ ] 함수 내에서 `let` 변수를 선언 후 재할당하는가?
- [ ] 로직의 정확성이 실행 순서에 의존하는가?
- [ ] 콜백 내부에서 여러 단계의 변이가 발생하는가?
- [ ] 함수를 제거하거나 순서를 바꾸면 결과가 달라지는가?

하나라도 해당하면 절차적 사고가 포함된 코드로 판단한다.

**주의 — 흔한 착각 식별:**
`map`, `filter`, `reduce`를 사용했더라도 콜백 내부가 절차적 사고(단계별 변이, 복잡한 조건 분기)라면 "배열 메서드를 사용했지만 절차적인 코드"로 판단한다. 도구가 아니라 사고방식이 기준이다.

```typescript
// 착각: map을 썼지만 내부는 절차적
items.map((item) => {
  let price = item.basePrice;
  if (item.discount) {
    price = price * (1 - item.discount);
  }
  price = price * 1.1;
  return { ...item, finalPrice: Math.round(price * 100) / 100 };
});

// 선언적: 각 변환이 비즈니스 관계를 나타냄
items.map(applyDiscount).map(addTax).map(formatPrice).filter(hasValidPrice);
```

### 기준 2: 상태 모델링

가능한 상태들 간의 관계를 타입 수준에서 명확히 정의하고 있는지 판단한다. 독립적인 boolean 플래그 여러 개로 상태를 관리하면서 불가능한 조합이 타입 수준에서 허용되는 경우를 찾아 지적한다.

```typescript
// 절차적: 불가능한 상태 조합이 허용됨
const [data, setData] = useState<T | null>(null);
const [loading, setLoading] = useState(false);
const [error, setError] = useState<string | null>(null);

// 선언적: discriminated union으로 불가능한 상태 원천 차단
type State<T> =
  | { status: "idle" }
  | { status: "loading" }
  | { status: "success"; data: T }
  | { status: "error"; error: string };
```

### 기준 3: 추상화 레벨의 적절성 (상대성)

선언적/절차적은 절대적 구분이 아니라 추상화 레벨에 따른 상대적 개념이다. 해당 코드가 속한 레벨에서 적절한 접근법을 사용하고 있는지 판단한다.

| 레벨                      | 적절한 접근 | 이유                        |
| ------------------------- | ----------- | --------------------------- |
| 비즈니스 로직 / 상태 관리 | 선언적      | 도메인의 본질적 관계 표현   |
| UI 구조 (JSX)             | 선언적      | 데이터-UI 대응 관계 표현    |
| 데이터 변환 파이프라인    | 선언적      | 각 단계 독립적, 테스트 용이 |
| 인프라 / 성능 최적화      | 절차적 OK   | 효율성이 우선               |
| 복잡한 에러 처리 / 재시도 | 절차적 OK   | 세밀한 제어 필요            |

### 기준 4: 구조적 관계 표현 (JSX/컴포넌트)

**선언적 JSX 신호:**

- 데이터 배열과 UI 리스트의 1:1 대응이 직관적으로 보임
- 컴포넌트 계층이 도메인 구조를 반영 (예: `Order > OrderItem > ProductInfo`)
- 조건부 렌더링이 상태 관계를 표현 (`status === "error" && <ErrorView />`)

**절차적 JSX 신호:**

- `useEffect` 내에서 DOM을 직접 조작
- `ref`를 통한 명령형 포커스/스크롤 관리가 비즈니스 로직과 혼재
- 렌더링 로직 안에서 복잡한 데이터 가공 (인라인 `map` 내부의 절차적 변환)

### 기준 5: 함수 추상화의 품질

**좋은 함수명 (비즈니스 의도):**

- `applyDiscount`, `validateOrder`, `formatCurrency`
- 이름만으로 입출력 관계가 예측 가능

**나쁜 함수명 (기술적 구현):**

- `processData`, `handleClick`, `doStuff`, `helper`
- 내부를 읽어야만 무엇을 하는지 알 수 있음

**합성 가능성 판단:**

- 각 함수가 단일 변환을 담당하여 `pipe(fn1, fn2, fn3)` 형태로 조합 가능한가?
- 아니면 하나의 함수가 여러 책임을 가져 분리 없이는 재조합이 불가능한가?

## Step 4: 결과 출력

### 4-1. 요약 판정

파일(또는 주요 함수/컴포넌트)별로 한줄 판정을 테이블로 출력한다.

```
| 대상 | 판정 | 심각도 | 레벨 적절성 | 한줄 요약 |
|------|------|--------|------------|----------|
| useUserData | 선언적 | — | 적절 | 데이터 관계를 선언적으로 표현 |
| processItems | 절차적 | P1 | 개선 필요 | 비즈니스 로직이지만 단계별 변이에 집중 |
| retryWithBackoff | 절차적 | — | 적절 | 인프라 레벨로 절차적 접근이 자연스러움 |
| useFormState | 착각 | P0 | 개선 필요 | boolean 플래그로 불가능한 상태 허용 |
```

판정 분류:

- **선언적** — 관계적 사고가 잘 드러남
- **절차적** — 시간적 순서/상태 변이에 집중
- **혼합** — 선언적 구조 안에 절차적 세부사항이 혼재
- **착각** — 선언적 도구를 사용했지만 실제로는 절차적 사고

심각도 분류 (레벨 적절성이 "적절"이면 심각도 `—`):

| 심각도 | 의미 | 예시 |
|--------|------|------|
| **P0 (Critical)** | 타입 안전성 침해 / 불가능한 상태 허용 | 독립 boolean 플래그로 상태 관리, 불가능한 상태 조합이 타입에서 허용됨 |
| **P1 (High)** | 비즈니스 로직이 절차적으로 표현됨 | 도메인 로직에서 let 변이 체인, 중첩 조건 분기로 상태 변환 |
| **P2 (Medium)** | 선언적 개선 가능하나 가독성에 큰 문제 없음 | 체이닝 가능한 단순 변환, 함수명이 기술적 구현을 드러냄 |
| **P3 (Low)** | 스타일 수준 개선 또는 허용 가능한 절차적 코드 | 미세한 네이밍 개선, 인프라 코드에 가까운 절차적 패턴 |

### 4-2. 상세 분석

기준별로 구체적인 코드 위치(`파일:라인`)를 인용하며 분석한다:

- **현재 코드**가 왜 절차적/선언적인지 근거 제시
- 해당 코드가 속한 **추상화 레벨**에서 이 접근이 적절한지 판단

### 4-3. 개선 제안

개선이 필요한 부분에 대해 **심각도 순(P0 → P3)**으로 그룹핑하여 구체적인 before/after 코드 예시를 제시한다.

개선 제안 시 반드시 **왜 이 변경이 더 선언적인지** 근거를 함께 제시한다:

- "이 변경으로 불가능한 상태 조합이 타입 수준에서 차단됩니다"
- "각 변환 단계의 비즈니스 의도가 함수 이름에서 바로 읽힙니다"
- "데이터와 UI의 대응 관계가 코드 구조에서 직접 드러납니다"

단, 인프라 레벨의 절차적 코드에 대해 무리하게 선언적으로 바꾸라고 제안하지 않는다.

### 4-4. 종합 요약 (다중 파일 분석 시)

3개 이상의 파일을 분석한 경우, 개별 분석 이후 전체 트렌드를 요약한다.

1. **판정 분포**: 선언적/절차적/혼합/착각 각 판정의 파일(함수) 수와 비율
2. **반복 패턴**: 2회 이상 등장하는 공통 안티패턴 식별 (예: "3개 파일에서 독립 boolean 상태 관리 패턴 발견")
3. **Top 3 개선 권장**: 가장 임팩트가 큰 개선 사항 3가지를 심각도와 영향 범위 기준으로 선정

## Step 5: 저장 (--save 옵션 시)

`--save` 옵션이 있으면 분석 결과를 md 파일로 저장한다. 파일 상단에 메타 정보를 포함한다:

```markdown
# 선언적/절차적 코드 분석 리포트

- **분석일**: {날짜}
- **대상**: {파일 목록 또는 diff 범위}
- **분석 기준**: "선언적 프로그래밍에 대한 착각과 오해" 분석 프레임워크
- **심각도 분포**: P0: {n}건 / P1: {n}건 / P2: {n}건 / P3: {n}건

---

{요약 판정 테이블}

{상세 분석}

{개선 제안 (심각도순)}

{종합 요약 (다중 파일 시)}
```

## 분석 시 유의사항

- **"모든 것을 선언적으로" 강요하지 않는다.** 절차적 코드가 적절한 레벨이 있다.
- **도구가 아닌 사고방식을 본다.** `map`을 썼는지가 아니라 콜백 내부의 사고가 관계적인지 판단한다.
- **순환논법을 피한다.** "React를 쓰니까 선언적"이 아니라, 왜 그 코드가 관계를 표현하는지 구체적 근거를 든다.
- **상대성을 인정한다.** 같은 코드도 어떤 레벨에서 보느냐에 따라 평가가 달라질 수 있다.
